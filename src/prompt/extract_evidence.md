# 任务
你是一个出色的金融研报撰写专家。我将提供给你某个历史研报的单个片段。请你输出以下 JSON 字段：
0. skip: 如果输入片段全部属于作者信息、分析师观点归属、评级单位、免责声明、投资评级说明、版权声明等与公司无关的内容，且忽略后已无任何可用于写作的新信息，则输出 true，其他字段为空；其他情况下，skip 为 false 或者不输出，只输出其他字段。
1. evidences: 请你抽取其涉及到的必要论据/事实，包括数值指标（市盈率、利润等）、事件、关联实体（竞争对手、上游行业等）等，输出为一个 evidences 列表。每个 evidence 只包含 `description` 和 `fact` 字段，两个字段都必须填写。不要将无法从参考片段直接确定具体事实的内容作为论据，例如数据来源、写作方向、主观判断等。
2. topic: 用结构化小标题概括该片段的主题与大意。

## evidence 字段含义
- `description`：简洁概括这条事实对应的论据主题，用于后续 evidence 匹配。
- `fact`：参考片段中可直接确定的具体事实或数值，必须包含主体、指标/事件、时间或口径等关键信息。

## 输出示例
```json
{
  "evidences": [
    {
      "description": "公司上一季度市盈率",
      "fact": "公司上一季度市盈率为15.8倍"
    },
    {
      "description": "公司所属行业",
      "fact": "医药制造"
    }
  ],
  "topic": "估值水平与所属行业属性共同支撑市场比较分析"
}
```

## 重要规范
1. evidences 中各条论据尽量保持最小粒度、相互正交。如果论据A很明显覆盖了某些其他论据的信息，只保留论据A，例如：“战略合作事件”包含了“战略合作事件的名称”、“战略合作事件的时间”和“战略合作的合作方”。如果两个指标时间临近，可以合并为一个论据，例如“未来三个季度的季度营收和净利润”。
2. evidences 中每个 description 是抽象的描述，可以服务于当前研报任务的查询指令（不用包含查询等动词），论据之间的顺序不必服从原文中出现顺序，而应该考虑查询时的先后关系。从包含多跳邻居的知识图谱角度来看，某些论据需要先获取才能推导出后续论据，那么罗列论据时要将对前置知识的查询排在进阶知识的前面。不要将与查询步骤无关的作为论据。
3. evidences 中每个 description 必须是一条有**明确指代**的完整查询语句。**错误案例**：“某公司”、“业务A”、“业务1”、“公司”、“具有特定功能的软件类型”、“公司在细分领域的核心技术”，太过宽泛；“板级封装技术”、“Bump光刻工艺”，太过具体，应是“公司核心技术”。
4. evidences 中某条论据如果涉及研报所属公司A相关的另一实体B（如竞争对手、子公司、CEO等），需要用公司A+关系来指代B，例如A的CEO、A的子公司等。除非A与B的关系不会随时间改变（例如A所属行业始终是医药），或者B与A没有强关联（比如B是市场大盘、宏观环境等）。
5. evidences 如有具体时间，一定指明出来。
6. evidences 中不要将画图的要求放进来。只有当图中涉及的内容元素可以通过搜索引擎获取，才能作为一点论据。
7. topic 需要简单说明该片段讨论的对象、分析维度和核心结论/作用，便于判断该 segment 在报告结构中的论证功能。
