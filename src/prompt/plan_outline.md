# 任务
你是一个出色的金融研报撰写专家。我将提供给你某个历史研报的单个片段。请你模仿总结出潜在论据、写作模版、（若干方面）写作要求等，适用于当前要撰写的新研报。该片段如果有图片，已经转换为了文本形式的图片描述，列论据和模版时请忽略。

## 需要输出的字段
0. skip: 如果输入片段全部属于作者信息、分析师观点归属、评级单位、免责声明、投资评级说明、版权声明等与公司无关的内容，且忽略后已无任何可用于写作的新信息，则输出 true，其他字段为空；其他情况下，skip为 false 或者不输出，只输出其他字段。
1. evidences：抽象化描述该片段涉及的必要论据/事实（0～6个），包括数值指标（市盈率、利润等）、事件、关联实体（竞争对手、上游行业等）。不要将无法搜索得到的、或者不需要搜索的声明的作为论据，例如数据来源等。只保留真正必要、可检索、可支撑写作的论据。
2. template：为了说明论据和论点的写作顺序和逻辑关系，参考原文撰写一份写作模版。整个模版作为template。
   - 对于**非图表的正文**：适当将提及的现象、指标数值、观点结论、图片图表等模糊化为占位符，例如`市盈率`、`市场占比饼状图`、`评价`、`上一季度`，除非该知识不太会随时间变化；对于其他可复用的原始语句，可以继续沿用。
   - 对于**表格**只需要描述列名和索引形式即可。
   - 对于可能包含的图片描述或画图要求，请忽略，不要放在模版中（因为研报正文中不要需要给画图要求、画图提示）。
3. requirements：分多个方面概括其涉及的内容要素、叙述结构、语法语义、数据可视化要求、写作特点等，作为写作要求。
4. topic：用十个字以内抽象地概括该段落内容。

## 输出示例
```json
{
  "evidences": [
    "上一季度市盈率",
    "该公司所属行业",
    "行业竞争对手",
    "竞争对手的上游行业",
    "本年度市值波动情况"
  ],
  "template": "`总结近期市场表现`，`预估未来表现`。`上一季度`市盈率为`上一季度市盈率`%，其竞对公司`竞争对手名称`市盈率为`上季度竞对市盈率`。`本年度`市值波动约`波动幅度`%上下，预计`本季度`持续上涨至`本季度预计市值`。",
  "requirements": [
    "总-分结构",
    "需要根据相关指标预估下一季度市值",
    "市场占比需要用饼状图呈现",
    "行文详略得当"
  ],
  "topic": "市场表现分析与预测"
}
```

如果输入片段全部属于作者信息、分析师观点归属、评级单位、免责声明、投资评级说明、版权声明等与研报写作无关的内容，且忽略后已无任何可用于写作的新信息，则不要输出上述四个部分，改为只输出：
```json
{
  "skip": true,   
  "evidences": [],
  "template": "",
  "requirements": [],
  "topic": ""
}
```

## 重要规范
1. template段落格式服从参考片段，不要随意换行，不要额外将非列表的文本用列表嵌套。
2. evidence、template、requirement、topic需要适当**模糊化实体名称、特点、观点**等在其他时间下容易变化的具体信息，保证可以泛化到其他研报中。只要是根据信息判断出的分析语句，也应模糊化。例如模糊化增长、减少、强相关、带动、领先、看好等形容词或动词。
3. 片段中如果出现作者信息、分析师观点归属、评级单位、免责声明等与研报写作无关的内容，直接忽略，不得写入evidence、template、requirement、topic的任何一个部分中。
4. evidence中只保留必要论据，禁止堆砌无检索价值或无写作必要的信息。
5. evidence中每条论据是可以服务于当前研报任务的查询指令（不用包含查询等动词），论据之间的顺序不必服从原文中出现顺序，而应该考虑查询时的先后关系。从包含多跳邻居的知识图谱角度来看，某些论据需要先获取才能推导出后续论据，那么罗列论据时要将对前置知识的查询排在进阶知识的前面。不要将与查询步骤无关的作为论据，例如"数据来源"、"图表风格白底简洁背景"等应列入写作要求。
6. evidence中各条论据尽量相互正交，总数不超过6条。如果两个论据极大概率被同时提及，合并在一起，例如"近两年利润变化和承压原因"。如果论据A很明显覆盖了某些其他论据的信息，只保留论据A，例如："战略合作事件"包含了"战略合作事件的名称"、"战略合作事件的时间"和"战略合作的合作方"，只需要查询"战略合作事件"即可得到相关信息。如果两个指标时间临近，可以合并为一个论据，例如"未来三个季度的季度营收和净利润"。
7. evidence中每条论据必须是一条有**明确指代**的完整查询语句，因为后续搜索时会将单独将一条论据送入搜索引擎，不会联系上下文。需要确保各行**没有**笼统模糊的代词，除非一定需要泛指，但即便泛指也不需要包括序号。**错误案例**："某公司"、"业务A"、"业务1"、"公司"、"具有特定功能的软件类型"、"公司在细分领域的核心技术"，太过宽泛，仅根据当前论据难以联想得到或者搜索获取。
8. evidence中某条论据如果涉及研报所属公司A相关的另一实体B（如竞争对手、子公司、CEO等），需要用公司A+关系来指代B，例如A的CEO、A的子公司等，确保可以泛化到其他任务。除非A与B的关系不会随时间改变（例如A所属行业始终是医药），或者B与A没有强关联（比如B是市场大盘、宏观环境等）。
9. evidence所涉及的数据需要指明时间范围，并且是相对于该研报撰写日期的相对时间，保证可以泛化到当前写作时间。除非特殊时间（如2018年成立、2023年股灾等），否则不能直接套用原文时间，错误案例："2025年第二季度"，正确案例："上一季度"。也不能过于模糊，错误案例："近几年"、"未来数年"、"未来X-Y年内"，正确案例："近两年"、"未来一年"。
10. evidence中不要将画图的要求放进来（应该放进requirement）。只有当图中涉及的内容元素可以通过搜索引擎获取，才能作为一点论据。
11. 请严格基于输入研报的片段提炼，不补充片段外信息。不要根据常识、行业印象或联想，自行加入原文未出现或未被直接需要的公司、机构、人物、产品等。
12. 如果原文中有表格，template中**一定不要放一个示例表**，**一定不要放表中示例数据**，这会导致文本过长。只需描述表头和索引列，再加上相应解读。
13. requirement还应总结参考句段的大致字数、是否包含深度分析等，反映撰写某要点时行文详略程度。
14. topic不是论据或论点的总结，不包含例如"显著"、"优异"之类的具体形容、描述、观点（错误案例：高增长业务推升整体毛利率、市场占比整体呈下降趋势等），而应表示该要点主要针对什么方面、维度（正确案例：业务增长情况和影响、近年市场占比变化）。
15. 你的输出结果必须与原片段的信息密度、篇幅层级、展开程度保持一致。**禁止把原文只有一两句话的内容扩展成长篇分析，不要补出原文未体现的分析层次**。
